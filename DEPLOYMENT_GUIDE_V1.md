# School Connect v1 — Detailed Deployment Guide

This guide covers two deployment tracks:
1. Deploying the generator itself
2. Deploying a school portal generated by the generator

All steps use free tools only.

---

# Part A — Deploy the Generator

## What you need
- a GitHub account
- optionally a Vercel, Netlify, or Cloudflare Pages account
- the `school connect v1` folder contents

## Option 1 — GitHub repository + GitHub Pages
1. Create a new GitHub repository.
2. Upload every file and folder from `school connect v1` exactly as they are.
3. Ensure the folder structure remains intact:
   - `assets/`
   - `database/`
   - root HTML files
   - markdown docs
   - `manifest.json`
   - `sw.js`
   - `.nojekyll`
4. Open the repository settings.
5. Go to **Pages**.
6. Set source to **Deploy from branch**.
7. Select branch `main` and folder `/root`.
8. Save and wait for publishing.
9. Open the published URL and test:
   - `index.html`
   - `builder.html`
   - `guide.html`
   - `cbt.html`
   - `notifications.html`

## Option 2 — Vercel static deployment
1. Log in to Vercel.
2. Import the GitHub repository.
3. Choose framework preset: **Other / Static**.
4. Leave build command empty.
5. Leave output directory as root or default static handling.
6. Deploy.
7. Visit the deployed site and test the same key pages.

## Option 3 — Netlify / Cloudflare Pages
1. Connect the GitHub repository or drag/drop the folder.
2. Do not use a build command.
3. Publish as static files.
4. Confirm page routing and assets load.

# Part B — Deploy a Generated School Portal

## Step 1 — Create a free Supabase project
1. Go to https://supabase.com
2. Sign in.
3. Create a new project.
4. Set a project name.
5. Generate and save a database password.
6. Choose the nearest region.
7. Wait for project setup to complete.

## Step 2 — Run the database schema
1. Open the Supabase dashboard.
2. Go to **SQL Editor**.
3. Create a new query.
4. Open the generated portal's `database/schema.sql` file.
5. Copy the full file contents.
6. Paste into SQL Editor and run.
7. If your project requires additional module SQL files, run them only in the exact order documented in that release package.
8. Confirm success messages and no errors.

## Step 3 — Get Supabase API values
1. Open **Project Settings**.
2. Open **API**.
3. Copy:
   - Project URL
   - anon public key

## Step 4 — Insert the keys into the generated school site
1. Open `assets/js/config.js` in the generated school portal.
2. Replace placeholder values with the real project URL and anon key.
3. Save the file.
4. Re-upload or re-commit if hosted through GitHub/Vercel/Netlify.

## Step 5 — Upload generated school site to hosting
1. Create a fresh GitHub repository for the school portal.
2. Upload the generated school site files exactly as downloaded.
3. Preserve structure and hidden files.
4. Enable GitHub Pages or deploy to Vercel/Netlify as static hosting.

## Step 6 — Create the first admin account
1. Open the deployed portal.
2. Use the request access / sign-up flow.
3. Go to Supabase SQL Editor.
4. Approve/elevate that first account using the documented SQL snippet for admin setup.
5. Log back in as the admin.

## Step 7 — Configure academic structure
1. Create sessions, terms, classes, departments, and subjects.
2. Create or import students.
3. Create or import staff.
4. Link parents to children.
5. Test each role.

## Step 8 — Configure CBT
1. Open `cbt.html` as staff/admin.
2. Create an exam.
3. Upload question CSV.
4. Open the exam for students.
5. Copy/share the exam code.
6. Open `cbt-exam.html` and test the student flow.
7. Verify result persistence in database.

## Step 9 — QA checklist before client use
- sign in/out works
- role restrictions work
- parent sees only linked child data
- student cannot open admin-only workflows
- staff can manage classroom academic pages
- CBT creation works
- CBT taking works
- calculator works during live exam
- scientific keyboard inserts symbols
- report card integration works
- notifications page loads
- PWA install prompt works

---

# Important warnings
- A visually working portal is not the same as a configured portal.
- If `config.js` still contains placeholder keys, database-driven features will not work.
- Role safety must be tested with real accounts, not only as a guest.
- The final client release should always be tested with at least one account in each major role:
  - admin
  - staff/teacher
  - parent
  - student

---

# Free-tool stack summary
- static HTML/CSS/JS hosting: GitHub Pages, Vercel, Netlify, Cloudflare Pages
- database/auth: Supabase free tier
- no paid AI APIs required
- no server infrastructure required for the basic version
